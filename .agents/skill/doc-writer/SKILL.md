# 文档编写器

为 InterX 子包生成全面的中文项目文档，附带可复现的裸机部署说明。

## 使用场景

- 用户要求为某个包或整个项目编写文档 / README / 技术文档
- 用户需要部署或环境配置说明
- 用户提到"运行说明"、"环境配置"、"依赖清单"等

## 工作流程

### 第一阶段：扫描

对目标包运行依赖扫描脚本：

```bash
python3 scripts/scan_deps.py <target_path>
```

若目标是整个项目，逐个包运行或对仓库根目录运行。

收集以下信息：
- 目录树（深度 3 层）
- 入口文件：`__main__.py`、`app.py`、`main.py`、`scripts/` 下的 CLI 脚本
- 配置文件：`.env.example`、`config.py`、`settings.py`、YAML/JSON 配置
- 已有 README（若存在——读取以保留原始意图）
- 测试目录和测试运行器

### 第二阶段：分析

读取关键源码以理解：
- 模块职责和公共 API
- 模块间 / 包间的数据流
- 外部服务依赖（数据库、API、消息队列）
- 端口、接口、环境变量
- 启动顺序和入口点

不要逐文件阅读。重点读取：
1. 配置 / settings 模块
2. 入口脚本（`app.py`、`main.py`、`scripts/` 下的脚本）
3. 顶层 `__init__.py` 了解包结构
4. 已有 README 或文档

### 第三阶段：生成

读取 `references/doc-template.md` 获取输出结构。

遵循 `references/style-guide.md` 的写作规范。

填写模板的每一个章节。若某章节不适用，写"不适用"并简要说明原因。

**必填章节（不可跳过）：**
- 项目概述
- 前置要求
- 项目结构
- 依赖清单
- 环境搭建（逐步骤、可复制命令）
- 配置说明
- 运行方式
- 测试

**可选章节（相关时包含）：**
- 架构 / 数据流
- 接口说明
- 常见问题与排错

### 第四阶段：交付

呈现完成的文档。若存在已有 README，简要说明变更内容。

询问用户是否需要调整后再写入文件。

## 范围控制

- **单个包**（如"给 `process` 写文档"）：仅文档化该包，简要提及跨包依赖。
- **整个项目**（如"给整个 InterX 写文档"）：编写顶层 README 加各包章节，
  或生成一份统一文档，用清晰的包边界分隔。
- 默认范围：用户指定的范围。若有歧义，询问用户。

## 规则

- 所有文档使用中文编写。
- 代码块中的注释也使用中文。
- 每条 shell 命令必须放在 ` ```bash ` 代码块中——可直接复制粘贴。
- 使用绝对路径或项目相对路径；不使用未定义的 `$HOME`。
- 已知时标注依赖版本约束（如 `fastapi>=0.100,<1.0`）。
- 始终以"全新干净机器"为前提编写步骤，假定无任何预先配置。
- 保留已有 README 的意图——增补而非丢弃。
- 不编造功能。若代码缺失，标注为"计划中"或"尚未实现"。
