# 文档编写风格指南

## 语言

- 所有文档使用中文编写。
- 技术术语保留英文原文（如 API、SDK、endpoint、pipeline 等）。
- 中英文混排时，英文单词前后加一个半角空格（如：使用 `pip install` 安装依赖）。
- 句子简短直接，多用主动语态。

## 语气

- 技术性强但易读。面向初次接触项目的开发者。
- 精确表述，避免模糊用词（如"等等"、"之类的"少用）。
- 不过度解释显而易见的概念。

## 标题

- 标题使用中文，简洁明了。
- 标题层级：`#` 用于标题，`##` 用于主要章节，`###` 用于子章节。
- 标题末尾不加标点。

## 代码块

- 始终指定语言：` ```bash `、` ```python `、` ```yaml `。
- 每个 bash 代码块可直接复制粘贴——不加 `$` 前缀。
- 代码块内的注释使用中文说明非显而易见的步骤。
- 代码块保持聚焦，每个代码块一个逻辑操作。

## 表格

- 结构化数据使用表格：依赖列表、环境变量、API 接口。
- 列对齐以保证可读性。
- 空单元格用"—"，"不适用"用于整个章节。

## 路径与命令

- 路径、命令、文件名使用行内代码：`scripts/build.py`。
- 优先使用项目相对路径，必须时才用绝对路径。
- 使用变量前先定义：先设 `PROJECT_ROOT=/path/to/InterX`。

## 版本约束

- 依赖标注主版本约束：`fastapi>=0.100,<1.0`。
- 若从 lock 文件获知确切版本，使用 `==`。
- 注明版本约束是建议还是硬性要求。

## 交叉引用

- 引用其他 InterX 包时用反引号：`retrieval`、`gateway`。
- 尽量使用相对路径链接到其他包的 README。

## 架构图

- 使用 ASCII 图描述架构和数据流。
- 图保持简洁——最多 10 个节点。
- 箭头标注数据类型或消息名称。

## InterX 专有术语

| 术语 | 定义 |
|------|------|
| big chunk | 大文档段落（约 2–4 页） |
| mid chunk | 中等段落（约 1 段话） |
| small chunk | 细粒度段落（约 1–2 句） |
| manual | `data/` 目录下的源 Markdown 文档 |
| manifest | 描述 chunk 关系的 JSON 索引文件 |
| gateway | 基于 LiteLLM 的 LLM 代理层 |
| KG | 知识图谱层（Neo4j / Kùzu） |
