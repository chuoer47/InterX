# 五重分词效果验证报告

## 实验数据概览

对比两种 BM25 分词策略在 350 条问答对上的检索召回率：
- **纯 jieba**：ASCII Token + jieba 分词（传统方案）
- **五重分词**：ASCII Token + jieba + unigram + bigram + trigram（InterX 方案）

语料：7,293 个 small chunks，来自 40 本中英文产品手册。

## 指标

| 指标 | 定义 |
|------|------|
| Recall@K | ground truth chunk_ids 出现在 BM25 top-K 中的比例 |
| 构建耗时 | BM25 索引构建时间 |
| 语料 token 数 | 分词后的总 token 数量（影响内存占用） |

## 实验开展

1. 加载 7,293 个 small chunks 的 `retrieval_text`
2. 分别用纯 jieba 和五重分词构建 BM25Okapi 索引
3. 对 350 条问题运行 BM25 检索（top-20）
4. 与 ground truth chunk_ids 对比计算 Recall@K

## 结果

| 指标 | 纯 jieba | 五重分词 | 提升 |
|------|---------|---------|------|
| 语料 token 数 | 303,833 | 855,302 | +181% |
| 构建耗时 | 0.3s | 1.3s | +1.0s |
| **Recall@5** | **9.9%** | **95.9%** | **+86.0%** |
| **Recall@10** | **9.9%** | **96.5%** | **+86.5%** |
| **Recall@20** | **9.9%** | **97.7%** | **+87.7%** |

- 五重分词独有命中：149 条
- 纯 jieba 独有命中：2 条

## 分析

1. **五重分词效果极其显著**：Recall@5 从 9.9% 提升到 95.9%，提升 +86 个百分点。这不是微小的优化，而是从"几乎不可用"到"优秀"的质变
2. **纯 jieba 召回率极低的原因**：产品手册中的中文术语（如"化油器调节螺钉"、"空气滤网"）被 jieba 错误切分后，查询词和文档词无法匹配。纯 jieba 依赖词典，对产品特有术语覆盖不足
3. **bigram/trigram 是关键**：五重分词中的 bigram 和 trigram 层提供了"模糊匹配"能力。即使 jieba 把"化油器"切成"化油"+"器"，trigram "化油器"仍然能匹配到文档中的同一术语
4. **token 数增加 181%**：五重分词使语料 token 数从 30 万增加到 85 万，但这只影响内存占用（约增加 2-3 倍），对检索延迟影响很小（BM25 打分是 O(n) 的简单计算）
5. **构建耗时增加 1 秒**：从 0.3s 到 1.3s，一次性成本，可忽略
6. **纯 jieba 有 2 条独有命中**：这两条可能是 jieba 分词恰好匹配了某个中文词汇，而五重分词因为 token 数量增加导致 BM25 的文档长度归一化略有不同。这是统计噪声，不影响结论

结论：五重分词是 InterX 检索系统的核心创新之一，将 BM25 召回率从 9.9% 提升到 95.9%，是系统可用性的关键保障。
