## 总目标

- 每个 chunk 应该是一个完整的语义单元，agent 不需要翻前后文也能知道这段在讲什么、适用于什么操作。
- chunk 之间使用且只使用一个空行分隔：

```text
<chunk1>

<chunk2>
```

- 不要为了追求长度均匀，把同一个操作、同一张图、同一个警告拆散。
- 原始 Markdown 的标题、步骤、图片、表格、注意事项应尽量保留原有顺序。

## 推荐切分单位

- 一个一级或二级标题下的完整说明，可以作为一个 chunk。
- 一个操作流程应整体作为一个 chunk，例如：安装电池、装入胶片盒、拍摄、回放、打印、插入存储卡。
- 一个界面说明应整体作为一个 chunk，例如：LCD 显示、选择按钮、命令转盘、菜单项目。
- 一个功能说明应整体作为一个 chunk，例如：滤镜、亮度调节、暗角调节、打印模式、快门按钮设置。
- 安全警告、注意事项、故障排除、规格参数、售后信息等独立栏目，可以各自作为 chunk。

## 图片处理原则

- 图片必须跟随它正在解释的文字，不能单独漂到前后相邻的 chunk。
- 如果图片展示的是某个步骤，图片应和该步骤、步骤前提、步骤后的注意事项放在同一个 chunk。
- 如果图片是部件名称、屏幕说明、菜单说明这类参考图，应和对应标题及说明文字放在同一个 chunk。
- 不要生成新的视觉说明文字；不要恢复旧版中以 `>` 开头的视觉模型描述。
- 不要新增图片文件。特别禁止把 PDF 渲染页、页面截图、裁剪图或模型生成图放入 `插图/` 目录来弥补原图缺失。
- Markdown 只能引用比赛允许展示的原始图片。若当前 Markdown 引用了后来新增的 `插图/<手册名>/page_*.png` 等文件，必须先把引用替换为合法原图或移除引用，再删除多余图片文件。
- `.codex/rendered/` 中的页面截图是校对材料，不是作品素材；这些文件不得出现在最终 Markdown 图片引用中。

## 步骤和注意事项

- 编号步骤不要拆开，除非原文中存在明确的子标题，且每个子流程能独立执行。
- “注意”“警告”“提示”“请确认”“如果……”这类文字必须贴近它约束的操作。
- 操作前提、操作步骤、操作结果应放在一起。例如“按下按钮”“屏幕显示结果”“打印完成”不能分在不同 chunk。
- 表格不要按行切开；一个表格应整体保留在同一个 chunk。

## 过长内容的拆分方法

- 优先按自然小标题拆分。
- 如果一个大标题下面有多个独立功能，可以按功能拆分，但每个 chunk 开头要保留足够的标题上下文。
- 如果拆出来的 chunk 会让人不知道它属于哪个模式或菜单，应在 chunk 内保留上级标题。
- 长度不是第一目标；语义完整优先。

## 不应该出现的切法

- 只有标题，没有正文。
- 只有图片，没有说明。
- 图片和对应步骤分离。
- 警告或注意事项和它约束的操作分离。
- 一个编号流程被随意截断。
- 表格被拆成多个 chunk。
- 为了补充上下文而新增未经原手册支持的解释。

## 切分前后检查

- 检查 chunk 之间是否正好一个空行。
- 检查每个 chunk 是否有明确主题。
- 检查每个图片是否在正确的说明附近。
- 检查是否还残留 `>`、`＞` 或 `&gt;` 开头的旧视觉说明。
