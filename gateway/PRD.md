# InterX Gateway PRD

## 1. 背景

InterX 计划采用多仓架构。当前首先建设一个独立的网关层（Gateway），作为后续多模态智能体客服系统的统一模型访问入口。该网关层优先基于 LiteLLM 搭建，用于统一接入多家中转站 / 模型提供方，并沉淀路由、缓存、监控、流式输出、可观测性与后续策略扩展能力。

当前阶段目标不是一次性完成全部能力，而是先形成一份可执行、可审核、可迭代的产品需求定义与技术路线，确保后续实现方向一致。

---

## 2. 当前目标

本阶段目标：

1. 创建 InterX 独立网关层目录与规划文档。
2. 明确基于 LiteLLM 的网关层职责边界。
3. 将现阶段模糊需求整理为可执行 PRD。
4. 给出第一版技术路线，待你审核后再开始实现。

本阶段**不直接开始编码实现业务功能**，除非你审核通过技术路线。

---

## 3. 网关层定位

InterX Gateway 是一个独立服务，职责包括：

- 统一封装上游 LLM / VLM / Embedding 提供方访问。
- 对外提供统一 API 入口，优先兼容 OpenAI 风格接口。
- 管理多中转站与多模型配置。
- 承担基础流量治理：路由、失败切换、缓存、限流、观测。
- 为后续 Agent / RAG / 多模态业务仓提供稳定底层能力。

非目标：

- 本阶段不实现完整客服 Agent 业务编排。
- 本阶段不实现复杂工作流引擎。
- 本阶段不承载业务知识库逻辑。
- 本阶段不实现最终用户侧产品前端。

## 4. 核心需求梳理

### 4.1 核心功能 1：多中转站接入 + latency-based routing

补充现实约束：

- 某些中转站在静态探测时可用，但在持续路由或容器网络环境下可能不稳定。
- 第一版需要支持“可用优先”策略：当 latency-based routing 条件不足或节点不稳定时，允许先退化为稳定节点优先 + 失败切换。
- 需要保留中转站探测、可用性打分、黑名单/冷却期机制，为后续更严格的 latency-based routing 做准备。


你后续会提供多个中转站。Gateway 需要支持：

- 配置多个上游站点 / API Base URL / API Key / 模型映射。
- 对同类模型请求进行路由分发。
- 支持基于延迟表现的路由策略（latency-based routing）。
- 在某个上游变慢、失败或不稳定时，能够自动切换到更优节点。
- 为后续扩展权重路由、成本优先路由、健康检查路由预留能力。

该能力优先依赖 LiteLLM 已有路由机制，不从零自研。

### 4.2 核心功能 2：关键指标检测 + 接入成熟监控前端

你希望“利用现有的，进行关键指标检测”，并尽量接入成熟的基于 LiteLLM 的资源检测 / 监控前端。

需求拆解如下：

- 采集网关层关键指标：
  - 请求量
  - 成功率 / 失败率
  - 响应延迟
  - 上游节点健康度
  - 模型级调用分布
  - Token 消耗
  - 缓存命中率
  - 流式请求占比
- 优先使用 LiteLLM 现成的可观测能力、代理 UI、日志 / 指标导出能力。
- 如 LiteLLM 原生能力不足，则通过 Prometheus + Grafana 等成熟组件补足。
- 最终希望集成一个“可直接查看”的前端/面板，而不是只有日志。

### 4.3 核心功能 3：支持 Stream 流式输出

Gateway 必须支持流式输出能力：

- 对外支持 OpenAI 风格 streaming。
- 路由、代理、缓存逻辑不能破坏 stream 能力。
- 后续上层 Agent / 前端可以直接消费流式返回。
- 需要明确哪些缓存路径支持 stream、哪些只支持非 stream。

### 4.4 核心功能 4：LiteLLM 缓存 + 混合缓存策略 + 小模型语义判定

你希望使用 LiteLLM 缓存能力，同时加入混合缓存策略：

- 先利用 LiteLLM 原生缓存能力。
- 引入多层缓存策略（精确匹配 + 语义缓存）。
- 对于“是否可视为同一问题”的判断，不单纯依赖 embedding 相似度。
- 增加一个小模型判定步骤：
  - 输入当前请求与候选缓存请求
  - 由小模型判断二者是否语义等价 / 是否可复用答案
  - 若小模型返回“是”，则视为缓存命中
- 该语义命中逻辑需要考虑：
  - 模型名
  - system prompt
  - 工具配置
  - 输入模态（文本 / 图片）
  - 业务上下文隔离
- 必须确保缓存命中不会造成高风险错误回答串用。

## 6. 使用场景

### 场景 A：普通聊天请求

上层服务发起一次标准 chat completion 请求，Gateway 根据模型别名找到对应提供方并返回结果。

### 场景 B：流式客服回答

上层服务发起 stream 请求，Gateway 将上游流式 token 持续透传给调用方。

### 场景 C：多中转站自动择优

同一个模型可挂多个中转站，Gateway 根据延迟表现自动选择更优节点。

### 场景 D：缓存命中复用

对于重复问题或高度语义等价问题，Gateway 命中缓存并直接返回结果，降低成本与延迟。

### 场景 E：运维监控

开发者或运维人员通过监控面板查看延迟、错误率、缓存命中、节点健康状况等。

- 完整审计后台
- 多区域部署
- 自动弹性扩缩容

## 8. 非功能需求

### 8.1 可扩展性

- 后续可平滑扩展更多模型提供方。
- 后续可支持文本、图像、多模态输入。
- 后续可挂接业务鉴权、中间件、审计、AB 测试。

### 8.2 可观测性

- 关键指标可视化。
- 日志结构化。
- 请求级问题可排查。
- 可区分上游问题、网关问题、缓存问题。

### 8.3 可维护性

- 配置驱动。
- 本地开发 / 测试 / 生产环境可分离。
- 文档先行。

### 8.4 风险控制

- 缓存命中策略要保守。
- 高风险请求默认不做激进语义缓存。
- 上游异常时要可降级、可切换、可定位。

---

## 9. 关键设计约束

- 优先复用 LiteLLM 官方能力，不重复造轮子。
- 如需扩展语义缓存逻辑，优先通过 LiteLLM 外挂层 / 前置层 / 中间件方式实现，而不是魔改源码。
- 对外接口优先兼容 OpenAI 风格，便于上层统一接入。
- 监控面板优先选成熟现成方案。

## 10. 待确认问题

以下问题需要在技术方案确定前与你确认：

1. 中转站是否都提供 OpenAI-compatible API？
2. 上游是否既包含文本模型，也包含视觉/多模态模型、embedding 模型？
3. 缓存是否需要跨模型共享，还是严格按模型隔离？
4. 小模型语义判定是否允许额外增加一次低成本模型调用？
5. 对高实时性的 stream 请求，是否允许先查缓存再决定是否直出？
6. 监控面板更偏好：
   - LiteLLM 原生 Admin / UI
   - Prometheus + Grafana
   - 二者同时保留
7. 是否需要从第一版开始就支持 API Key 鉴权和访问审计？
